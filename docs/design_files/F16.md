# F16: Save Analyzed Games in Memory (Redis Cache)

## Overview
To improve read performance and reduce the database load on MongoDB, completed game analyses should be cached in Redis with a 1-day Time-To-Live (TTL). When a user requests the analysis of a game, the API gateway should check Redis first. If found, it displays the analysis directly from Redis. Otherwise, it retrieves the analysis from MongoDB, writes it to Redis (read-through caching), and returns it. Additionally, when the C worker completes analyzing a game, it should proactively write the analysis to Redis with a 1-day TTL (write-through caching).

## Cache Key and Structure
- **Key**: `game:${gameId}:analysis`
- **TTL**: 1 day (`86400` seconds)
- **Value**: A serialized JSON string of the completed analysis:
  ```json
  {
    "moves": [...],
    "playerSummaries": [...]
  }
  ```

## Detailed Changes

### 1. C Worker (`src/analyze-service/src/worker.c`)
In `run_game_analysis` (after JSON generation and database updates), write the completed analysis to Redis.

**Proposed Change**:
```c
// Save analyzed game to Redis with 1-day (86400s) TTL
redisReply *cache_set = redisCommand(redis, "SET game:%s:analysis %s EX 86400", game_id, analysis_json);
if (cache_set) {
    freeReplyObject(cache_set);
}
```

---

### 2. API Gateway Route (`src/api-gateway/src/routes/games.ts`)
Update `GET /api/games/:gameId/analysis` to implement the read-through cache check.

**Proposed Logic**:
1. Check Redis for `game:${gameId}:analysis`.
2. If hit:
   - Parse and return the cached JSON object formatted as:
     ```json
     {
       "gameId": gameId,
       "moves": parsed.moves,
       "playerSummaries": parsed.playerSummaries
     }
     ```
3. If miss:
   - Query MongoDB for the game document.
   - If the game is not found, return `404`.
   - If the status is `failed`, return `500`.
   - If the status is `completed`:
     - Construct the cached analysis payload `{"moves": game.analysis.moves, "playerSummaries": game.analysis.playerSummaries}`.
     - Write it to Redis with a 1-day TTL using `redis.set(key, JSON.stringify(payload), 'EX', 86400)`.
     - Return the response.
   - If status is `queued` or `running`, fall back to the live progress polling/409 response.
