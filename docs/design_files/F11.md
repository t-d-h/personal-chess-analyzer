# F11: Support for Chess.com Analysis Review URLs

## Overview
Users want to analyze games using URLs copied from the Chess.com analysis board or review page, which look like `https://www.chess.com/analysis/game/live/{id}/review`. 
Currently, the frontend blocks these because it strictly expects `.includes('chess.com/game/')`.

## Changes Required

### Frontend
- **File**: `frontend/src/components/InputForm.tsx`
- **Change**: Update the URL validation logic so that it accepts URLs containing either `chess.com/game/` or `chess.com/analysis/game/`. This applies to both the explicit URL input field validation and the PGN fallback heuristic that treats strings as URLs if they match the domain.

### Backend
- **File**: `src/api-gateway/src/services/chesscom.ts`
- **Verification**: The current regex `/^https?:\/\/(?:www\.)?chess\.com\/(?:analysis\/)?game\/(?:live|daily)\/([^\/?#]+)/i` natively supports `analysis/` as an optional path and extracts the game ID up until the `/review` segment because of the `[^\/?#]+` character class.
- **Change**: No functional change is necessary, but the logic should be validated with automated tests to guarantee it handles the trailing `/review` correctly.

### Testing
- **File**: `tests/test_chesscom.py`
  - Add a test `test_valid_chesscom_review_url` simulating a POST to `/api/games` using a complete review URL: `https://www.chess.com/analysis/game/live/test-valid-mock/review`. Ensure it resolves the ID correctly.
- **File**: `tests/test_frontend_e2e.py`
  - Add or update an end-to-end test submitting the analysis review URL variant to ensure the entire flow works from browser to database.
