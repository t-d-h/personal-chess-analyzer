# F06 — Queue Integration (C analyze-service, Redis + MongoDB)

## Behavior
The C analyze-service runs as a long-lived worker process: it reads jobs from the Redis stream via `XREADGROUP`, processes each game using F05's engine loop, writes per-move progress to a Redis hash after each move, and writes the final analysis document to MongoDB on completion. `XACK` is called only after a successful MongoDB write.

## Scope
- C analyze-service only.
- Full end-to-end: Redis → Stockfish → MongoDB.
- API gateway is unchanged.
- Verified by `pytest tests/test_analyze_service.py`.

## Worker architecture

### Process model
Single process, `WORKER_CONCURRENCY` threads (configurable via env var, default 2). Each thread independently:
1. Calls `XREADGROUP` with `COUNT 1 BLOCK 5000` — blocks up to 5s waiting for a job.
2. Processes the job (run analysis loop).
3. Writes to MongoDB.
4. Calls `XACK`.

Using pthreads — no async I/O needed since each thread is fully synchronous.

```c
// main_worker.c
int main() {
    RedisConn redis = redis_connect(getenv("REDIS_URL"));
    MongoConn mongo = mongo_connect(getenv("MONGO_URL"));
    int concurrency = atoi(getenv("WORKER_CONCURRENCY") ?: "2");

    pthread_t threads[MAX_WORKERS];
    WorkerCtx ctx = { .redis = &redis, .mongo = &mongo };
    for (int i = 0; i < concurrency; i++)
        pthread_create(&threads[i], NULL, worker_thread, &ctx);
    for (int i = 0; i < concurrency; i++)
        pthread_join(threads[i], NULL);
}
```

## Redis operations

### Consume job
```
XREADGROUP GROUP workers worker-<pid>-<tid> COUNT 1 BLOCK 5000 STREAMS chess:analysis-jobs >
```
Returns: `gameId`, `pgn` from the message fields.

### Progress update (after each move)
```
HSET job:<gameId>:progress status running movesAnalyzed <N> movesTotal <M>
EXPIRE job:<gameId>:progress 3600
```

### Completion
```
HSET job:<gameId>:progress status completed movesAnalyzed <M> movesTotal <M>
EXPIRE job:<gameId>:progress 300
```

### Failure
```
HSET job:<gameId>:progress status failed errorMessage "<msg>"
EXPIRE job:<gameId>:progress 300
```

### Acknowledge
```
XACK chess:analysis-jobs workers <entry-id>
```
Called only after MongoDB write succeeds (or after marking failed).

## MongoDB write

### On completion — update the game document
```js
db.games.updateOne(
  { _id: ObjectId(gameId) },
  {
    $set: {
      "analysis.status": "completed",
      "analysis.movesAnalyzed": M,
      "analysis.completedAt": new Date(),
      "analysis.updatedAt": new Date(),
      "analysis.moves": [ ... ],
      "analysis.playerSummaries": [ ... ]
    }
  }
)
```

Use `libbson` to build the document and `libmongoc` to execute the update.

### On failure
```js
db.games.updateOne(
  { _id: ObjectId(gameId) },
  { $set: { "analysis.status": "failed", "analysis.errorMessage": "<msg>", "analysis.updatedAt": new Date() } }
)
```

## C libraries

| Library | Use |
|---------|-----|
| `hiredis` | Redis client (sync, one connection per thread) |

## Test plan (`tests/test_analyze_service.py`)
Python integration test using `docker compose up` (compose fixture in `conftest.py`).

| Test | Expectation |
|------|------------|
| Submit a game via POST /api/games, wait ≤30s | analysis.status = "completed" in MongoDB |
| Progress hash is updated during processing | HGET job:<id>:progress status = "running" mid-job |
| Progress hash has TTL after completion | TTL ≤300s |
| Correct move count in MongoDB | movesTotal matches PGN ply count |
| XACK called — stream has no pending entries | XPENDING count = 0 after completion |

## Retry / reclaim (basic v1)
On startup, each worker runs `XAUTOCLAIM` to reclaim stream entries idle for >10 minutes:
```
XAUTOCLAIM chess:analysis-jobs workers worker-<pid>-<tid> 600000 0-0 COUNT 10
```
Reclaimed entries are processed normally. No dead-letter handling in v1.
