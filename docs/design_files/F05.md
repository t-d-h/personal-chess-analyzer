# F05 — Full Game Analysis (C, synchronous, no queue)

## Behavior
The C analyze-service binary loops over every move in a PGN game using a single long-lived Stockfish subprocess, computes per-move win% loss, accuracy score, and move classification, and prints a correct move-by-move table. Correctness is verified against a known reference game with pre-computed expected values.

## Scope
- C only — extends F04's `uci.c` with `analyzer.c`.
- No Redis or MongoDB — reads PGN from stdin or a file, outputs JSON to stdout.
- Verified by `make test-analyze-game`.

## PGN parsing in C
Use a small custom PGN parser or embed a single-header library:
- **Preferred**: `sunfish`-style simple parser — walk the move list, apply each move to an internal board to derive FEN after each ply.
- Use `chess.js` logic ported to C is too complex; instead: link against `libchess` (MIT, single-file header) or implement a minimal FEN generator from move sequences.

**Recommended approach**: Call out to `node` or a small helper script to expand PGN → list of FENs (one per ply), then pipe those into the C analyzer. This avoids reimplementing a full chess engine board in C for v1.

### Two-phase design
```
pgn-to-fens <game.pgn>   → newline-separated FEN list (Node.js helper script using chess.js)
analyze-worker --fens -   → reads FENs from stdin, runs Stockfish on each, outputs JSON
```

The `make test-analyze-game` target pipes them together.

## Accuracy math (implemented in `analyzer.c`)

### win% conversion (from ARCHITECTURE §3.6)
```c
double win_percent(double cp) {
    // cp capped to [-10000, 10000] before calling
    return 50.0 + 50.0 * (2.0 / (1.0 + exp(-0.00368208 * cp)) - 1.0);
}
```

### win% loss
```c
double wp_loss = fmax(0.0, wp_best - wp_played);
```

### move accuracy score
```c
double move_accuracy(double wp_loss) {
    double raw = 103.1668 * exp(-0.04354 * wp_loss) - 3.1669;
    return fmax(0.0, fmin(100.0, raw));
}
```

### game accuracy (per player)
Average of `move_accuracy` across that player's moves.

## Move classification logic
```c
typedef enum {
    BOOK, BEST, EXCELLENT, GOOD, INACCURACY, MISTAKE, BLUNDER, BRILLIANT
} Classification;

Classification classify(int cp_loss, int is_sacrifice, int is_best_move) {
    if (is_book) return BOOK;
    if (is_sacrifice && is_best_move) return BRILLIANT;
    if (cp_loss == 0) return BEST;
    if (cp_loss < 50) return EXCELLENT;
    if (cp_loss < 100) return GOOD;
    if (cp_loss < 300) return INACCURACY;
    if (cp_loss < 500) return MISTAKE;
    return BLUNDER;
}
```

**cp_loss** = `evalCpBest - evalCpPlayed` (both from the same side-to-move perspective).

**Sacrifice detection**: a move is a sacrifice if the material value of pieces on the destination square (captured piece) > 0 AND `multipv 2` score is meaningfully better (e.g. > 50cp) than a non-capturing alternative. For v1, simplify: a move is a sacrifice if the played move is a capture and the engine's 2nd-best move is non-capturing.

**Book move**: skip classification for the first `BOOK_PLIES=10` plies.

## Reference game for verification
Use a short (≤20 moves) famous game with known accuracy metrics. Example: Scholar's Mate game.

Store expected output in `analyze-service/tests/reference_game.pgn` and `reference_output.json`.

`make test-analyze-game` runs the pipeline and `diff`s the key numeric fields (accuracy to 1dp, cp to nearest 5cp).

## Makefile targets
```makefile
build-analyzer:
	$(CC) $(CFLAGS) src/uci.c src/analyzer.c src/main_game.c -o bin/analyze-game -lm

test-game:
	node tools/pgn_to_fens.js tests/reference_game.pgn | \
	  ./bin/analyze-game --fens - | \
	  python3 tests/check_output.py tests/reference_output.json
```
